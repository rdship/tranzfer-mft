## Section 1: Performance

| ID | Area | Issue | Evidence | Risk | Fix | Priority |
|----|------|-------|----------|------|-----|----------|
| P-01 | VFS Advisory Lock | `pg_advisory_xact_lock` hash computed as `(long) accountId.hashCode() * 31 + path.hashCode()`. Java `int.hashCode()` returns an `int`; after widening to `long` and multiplication, collisions between different `(accountId, path)` pairs map to the same lock slot. Two unrelated accounts writing to differently-hashed paths can spuriously serialize — each believing it holds the "same" lock. At N accounts * M paths, the 64-bit key space is addressed via 32-bit Java hash math, giving O(N*M / 2^32) collision probability — non-trivial at scale. | `VirtualFileSystem.java:550` | MED | Replace hash with `UUID.getMostSignificantBits() ^ UUID.getLeastSignificantBits() ^ (long) path.hashCode() * 0x9e3779b97f4a7c15L` (Fibonacci hashing). This distributes uniformly across the full `long` range. Add a unit test asserting no collisions for 10,000 (accountId, path) pairs. | P2 |
| P-02 | HikariCP Pool Sizing | `onboarding-api` pool=8, `screening-service` pool=3, `config-service` pool=5, `encryption-service` pool=3, `analytics-service` pool=5, `ai-engine` pool=3, `storage-manager` pool=5. | S-01 | Advisory Lock Hash Collision — DoS Vector | Same collision as P-01, viewed from a security angle: an adversary who can enumerate or craft VFS paths that collide with a target account's lock hash can perform a Denial-of-Service via lock contention. The attacker holds the advisory lock (by keeping a transaction open) and serializes all writes to the victim account's colliding paths. Lock contention is silent — victims see slow writes, not errors. | `VirtualFileSystem.java:550` | HIGH | Apply the Fibonacci hash fix from P-01. Additionally, assert in the `normalizePath` unit tests that paths cannot be crafted to collide with known account UUIDs. | S-02 | VFS Path Traversal — Incomplete Normalization | `VirtualFileSystem.normalizePath()` converts backslashes to forward slashes and collapses `//`, but does **not** resolve `..` components. An SFTP client sending the path `../../etc/passwd` (after URL-decoding) would normalize to `../../etc/passwd` (starts with `/` after prefix, but contains `..`). If any VFS caller does not pre-strip `..`, this escapes the virtual root into the DB-level path namespace. A path like `/accounts/UUID/../admin-uuid/secret.key` could access another account's files if account-scoping is done by path prefix rather than account UUID foreign key. | `VirtualFileSystem.java:587-600` | HIGH | Add to `normalizePath()`: resolve `..` components using `java.nio.file.Path.normalize()` (in-memory, no filesystem access). | M-03 | Docker Compose — No Resource Limits | `docker-compose.yml` does not set `mem_limit` or `cpus` on any service. Any service experiencing a memory leak, GC pressure spike, or runaway query can consume all container memory — causing the Docker daemon to OOM-kill neighboring services (typically the largest consumers: storage-manager, ai-engine, PostgreSQL). OOM kills leave intents PENDING and can corrupt in-flight transfers. | `docker-compose.yml` — all services lack `mem_limit` / `cpus` | HIGH | Add per-service resource limits matching the service profile: `onboarding-api: mem_limit: 768m, cpus: '1.0'`, `screening-service: mem_limit: 512m, cpus: '0.5'`, `ai-engine: mem_limit: 1g, cpus: '1.0'`, `storage-manager: mem_limit: 1g, cpus: '1.5'`, `postgres: mem_limit: 2g`. Set `memswap_limit` equal to `mem_limit` to prevent swap-induced latency. | P1 |
| M-04 | No Pre-Stop Hook for AMQP Consumer Drain | When a container receives `SIGTERM`, Spring Boot begins its graceful drain (30s window via `server.shutdown: graceful`).
