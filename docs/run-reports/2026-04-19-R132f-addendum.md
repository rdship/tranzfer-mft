## Why the fix didn't catch it

Dev's `awaitAvailable()` only helps when SPIFFE `isAvailable()` was
returning `false` during the boot race. In this env SPIFFE **is**
available — boot logs confirm:

```
[SPIFFE] Workload API connected — socket=unix:/run/spire/sockets/agent.sock
         identity=spiffe://filetransfer.io/config-service
```

So `awaitAvailable()` returns true immediately, a JWT-SVID is attached,
and the downstream **rejects the attached token** with 403. Dev's R132
batch-2 `BaseServiceClient.addInternalAuth` admin-JWT fallback only
fires when no token is attached — the fallback never triggers in this
failure mode.

The real failure is: **the JWT-SVID attached by onboarding-api /
config-service / flow-engine has an audience or scope claim that
encryption-service's / external-forwarder-service's
`PlatformJwtAuthFilter` does not accept**. Neither R132 nor R132f
touched the token factory nor the downstream filter.

## R133 ask — concrete

1. **Diagnose the token rejection.** Log at DEBUG on the downstream
   service the exact reason the JWT-SVID is rejected. Likely candidates:
   - audience mismatch (token `aud` = `onboarding-api`, filter expects
     `encryption-service`),
   - missing scope claim,
   - signature verification against wrong SPIRE trust bundle.
2. **Fix the audience.** Either make the token factory issue
   per-target-audience SVIDs, or relax the downstream filter to accept
   any SVID within the same trust domain.
3. **Add a secondary fallback in `BaseServiceClient`.** Even if the
   primary SPIFFE attach succeeds, if the downstream returns 401 /
   403, retry once with the admin-JWT from the inbound request
   context. Belt-and-braces for user-initiated calls (BUG 12).
4. **Separate pin per S2S hop.** Integration test that `POST` /api/delivery-endpoints
   with an ADMIN JWT resolves to 201, and that `FILE_DELIVERY` of a
   SFTP-uploaded file reaches an external SFTP mock (atmoz/sftp
   sidecar) byte-identically.
5. **Don't re-grade R132 until both hops work at runtime.** R132 🥈
   Silver stands only because the other 5 backlog items closed; BUG 13
   is a primary outbound feature that is still non-functional for any
   admin on any flow.

## Reference setup

The 3rd-party SFTP baseline that exposes BUG 13 is reproducible in
one command:

```
docker run -d --name thirdparty-sftp \
  --network tranzfer-mft_default \
  -v /tmp/thirdparty-sftp:/home/partner/upload \
  atmoz/sftp:alpine partner:PartnerPass123!:1001
```

Then create a flow with `FILE_DELIVERY` step referencing a
`delivery_endpoints` row pointing at `thirdparty-sftp:22`, SFTP-upload
a file matching the pattern, and watch the failure in
`flow_executions.error_message`. When BUG 13 is closed, bytes land at
`/tmp/thirdparty-sftp/upload/` on the host with byte-identical SHA-256
vs the source.
